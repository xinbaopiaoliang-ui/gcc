# 业务后台游戏配置字段补充说明

> 适用对象：业务后台开发、客户端开发、控制面板对接开发。
>
> 本文用于补充《游戏加速配置字段对接说明》。原文档偏数据库字段视角，可以保留；但节点和客户端联调需要再补一层“运行时转换规则”。业务后台不要把 `GM_GameZone` / `GM_AccNode` 原始数据库字段直接理解成“全部下发给节点”。

## 一、总判断

业务后台现有字段整体可以继续使用，但还缺少以下运行时信息：

1. 节点 `hmac_secret` 的生成、保存和同步规则。
2. 客户端短期 JWT token 的签发字段。
3. `GM_GameZone` 规则转换成节点 `route_policies` 的规则。
4. 客户端每次 `OPEN_TCP` / `OPEN_UDP` 必须携带的 `metadata`。
5. 控制面板节点同步所需的 `admin_host`、`ssh_host`、`desired_policy_revision` 等字段。

当前节点实际消费的是三类数据：

| 数据 | 来源 | 节点用途 |
| --- | --- | --- |
| 短期 JWT token | 业务后台签发 | 鉴权、用户、设备、连接数、限速、TCP/UDP 权限、游戏和策略授权 |
| `OPEN_TCP` / `OPEN_UDP` 请求 | 客户端发起 | 指定真实目标地址、端口和本次 flow 的业务元信息 |
| `route_policies` 策略快照 | 业务后台生成，经控制面板下发 | 校验 `game_id`、`policy_id`、`rule_id`、目标地址和端口是否允许 |

节点不会直接消费完整数据库字段，也不会执行客户端本机逻辑，例如进程扫描、hosts 写入、智能 DNS、清 hosts、伪 ping、启动命令、三网 IP 选择。

## 二、必须补充的字段和规则

### 1. 节点 HMAC Secret

业务后台需要为每个节点生成并保存一份随机 `hmac_secret` 明文。

| 字段 | 保存位置 | 用途 |
| --- | --- | --- |
| `hmac_secret` 明文 | 业务后台数据库 | 业务后台给客户端签发 JWT |
| `hmac_secret` 加密副本 | 控制面板数据库 | 一键部署节点时写入节点配置 |
| `auth.hmac_secret` 明文 | 节点 `/etc/gaccel-node/config.yaml` | 节点验证客户端 JWT |

注意：

- `hmac_secret` 是每个节点独立生成，不建议全平台统一一个。
- `hmac_secret` 不能下发给客户端。
- 控制面板登录 JWT 不能拿来调用业务后台接口，也不能当作节点 `hmac_secret` 使用。
- `backend_api_key` 只用于业务后台调用控制面板 `/api/backend/*`，不是客户端 token 签名密钥。

### 2. 客户端 token claims

业务后台给客户端签发短期 JWT 时，建议至少包含：

```json
{
  "sub": "user-10001",
  "user_id": "user-10001",
  "device_id": "win-device-01",
  "exp": 1782115200,
  "nbf": 1782111600,
  "iat": 1782111600,
  "max_connections": 64,
  "rate_limit_mbps": 100,
  "allow_tcp": true,
  "allow_udp": true,
  "game_ids": ["steam"],
  "policy_ids": ["steam-web-v1"],
  "config_revision": "20260622.1"
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `user_id` | 是 | 用户 ID |
| `device_id` | 是 | 设备 ID |
| `exp` | 是 | 过期时间，Unix 秒；节点会拒绝没有过期时间的 token |
| `max_connections` | 是 | 该 token 允许的最大 QUIC 会话数 |
| `rate_limit_mbps` | 是 | 用户限速 |
| `allow_tcp` | 是 | 是否允许 TCP flow |
| `allow_udp` | 是 | 是否允许 UDP flow |
| `game_ids` | 建议 | 用户被授权的游戏 ID 列表 |
| `policy_ids` | 建议 | 用户被授权的策略 ID 列表 |
| `config_revision` | 建议 | 客户端配置版本，必须和 flow metadata 对齐 |

业务后台可以调用控制面板：

```http
GET /api/backend/token-defaults
```

读取默认 `max_connections`、`rate_limit_mbps`、`allow_tcp`、`allow_udp`，再结合自己的套餐覆盖。

### 3. 控制面板节点同步字段

业务后台同步节点给控制面板时，建议补齐：

```json
{
  "node_id": "hk-01",
  "name": "香港 01",
  "region": "hk",
  "country": "HK",
  "provider": "aliyun",
  "line_type": "premium",
  "endpoint_host": "47.83.160.126",
  "endpoint_port": 6666,
  "alpn": "gaccel/1",
  "admin_host": "47.83.160.126",
  "admin_port": 5557,
  "ssh_host": "47.83.160.126",
  "ssh_port": 22,
  "ssh_user": "root",
  "allow_tcp": true,
  "allow_udp": true,
  "hmac_secret": "per-node-random-secret",
  "tags": ["steam", "quic"],
  "labels": {
    "line": "premium"
  },
  "status": "online",
  "desired_version": "",
  "desired_policy_revision": ""
}
```

容易混淆的字段：

| 字段 | 用途 |
| --- | --- |
| `endpoint_host` / `endpoint_port` | 客户端连接 QUIC 节点使用 |
| `admin_host` / `admin_port` | 控制面板访问节点 `/health`、`/status`、`/panel/commands` 使用 |
| `ssh_host` / `ssh_port` / `ssh_user` | 控制面板一键部署、一键更新、修复任务使用 |
| `desired_version` | 期望节点二进制版本 |
| `desired_policy_revision` | 期望节点应用的策略版本 |

## 三、数据库字段到运行时字段的映射

### 1. 游戏和区服

| 原字段 | 是否保留 | 运行时映射 |
| --- | --- | --- |
| `GM_GameID` | 是 | 转成 `game_id`，写入 token、客户端配置和节点策略 |
| `GM_GameZoneID` | 是 | 转成 `policy_id`，写入 token、客户端配置和节点策略 |
| `ZoneName` | 是 | 客户端展示，不发节点 |
| `Version` | 是 | 建议转成统一 `config_revision` / `route_policies.revision` |
| `ProcessRules.ProcessName` | 是 | 客户端本机进程匹配；命中后只把 `process_name` 放入 metadata |
| `ProcessRules.Sign` / `Md5` | 可选 | 客户端本机校验，不发节点 |
| `DomainRules` | 是 | 转换成 `route_policies.rules` |
| `IpRules` | 是 | 转换成 `route_policies.rules` |
| `Hosts` | 视客户端能力 | 客户端本机 hosts 操作，不发节点 |
| `SmartDns` | 视客户端能力 | 客户端本机 DNS 逻辑，不发节点 |
| `CmdExec` | 视客户端能力 | 客户端本机执行，不发节点 |
| `SkipMainlandIP` | 视客户端能力 | 客户端或后台预处理，不发节点 |
| `ClearHosts` | 视客户端能力 | 客户端本机操作，不发节点 |
| `FakePingValue` | 视客户端能力 | 客户端展示或 hook，不发节点 |

### 2. 加速节点

| 原字段 | 是否保留 | 运行时映射 |
| --- | --- | --- |
| `GM_AccNodeID` | 是 | 转成 `node_id`，必须按字符串处理 |
| `NodeName` | 是 | 客户端和面板展示 |
| `ServerIP` | 是 | 默认映射为 `endpoint_host` |
| `ServerPort` | 是 | 映射为 `endpoint_port` |
| `DisableQuic` | 是 | 当前节点协议就是 QUIC；为 true 时客户端应过滤该节点 |
| `Area` | 是 | 客户端展示或筛选，不发节点 |
| `BandwidthQuality` | 是 | 后台或客户端做线路筛选，不参与节点单次 flow 校验 |
| `IsSupportIpv6` | 可选 | 后台或客户端决定使用 IPv4/IPv6 入口 |
| `TelecomIP` / `MobileIP` / `UnicomIP` | 可选 | 后台或客户端按运营商选择最终 `endpoint_host` |
| `RelayServerIP` / `RelayServerPort` | 暂不接入 | 当前节点协议没有中继字段；需要单独设计 |
| `Status` | 是 | 禁用节点不要下发给客户端，不要签发 token |
| `Tag` / `Sort` | 可选 | 客户端筛选和排序 |

## 四、DomainRules / IpRules 转换成 route_policies

节点策略格式：

```yaml
route_policies:
  revision: "20260622.1"
  policies:
    - policy_id: "steam-web-v1"
      game_id: "steam"
      allow_tcp: true
      allow_udp: true
      rules:
        - rule_id: "steam-store-tcp-443"
          network: "tcp"
          target_type: "domain"
          target_value: "store.steampowered.com"
          port_start: 443
          port_end: 443
          action: "quic_relay"
          priority: 100
          enabled: true
```

### DomainRules 转换

| 原字段 | 节点字段 | 转换规则 |
| --- | --- | --- |
| `Protocol` | `network` | `["tcp","udp"]` 建议拆成两条规则 |
| `Domain` | `target_type` / `target_value` | 精确域名用 `domain`；`*.example.com` 转成 `domain_suffix` + `.example.com` |
| `Port` | `port_start` / `port_end` | 逗号和范围需要拆成多条规则 |
| `IsAcc=true` | `action=quic_relay` | 允许走节点 |
| `IsAcc=false` | `action=direct` 或客户端本地排除 | 不要命中后继续发起节点 `OPEN` |

当前节点支持的 `target_type`：

```text
domain
domain_suffix
ip
cidr
any
```

注意：当前节点不支持 `api*.example.com` 这种中间通配。业务后台要么拆成可表达的精确域名/后缀域名，要么后续提出节点 matcher 扩展需求。

### IpRules 转换

| 原字段 | 节点字段 | 转换规则 |
| --- | --- | --- |
| `Protocol` | `network` | `tcp` / `udp` / `any` |
| `IpCidr` | `target_type=cidr` / `target_value` | CIDR 可直接使用 |
| `Port` | `port_start` / `port_end` | 逗号和范围需要拆成多条规则 |
| `IsAcc=true` | `action=quic_relay` | 允许走节点 |
| `IsAcc=false` | `action=direct` 或客户端本地排除 | 不要命中后继续发起节点 `OPEN` |

端口规则：

- 节点策略端口范围必须是 `1-65535`。
- 原文档里的 `"0-65535"` 必须转换成 `"1-65535"`。
- `"80,443,5000-6000"` 应拆成三条规则：`80-80`、`443-443`、`5000-6000`。

## 五、客户端每次连接节点时必须带的 metadata

客户端命中规则后，不是把完整游戏配置发给节点，而是发 `OPEN_TCP` 或 `OPEN_UDP`。

TCP 示例：

```json
{
  "type": "OPEN_TCP",
  "target_host": "store.steampowered.com",
  "target_port": 443,
  "metadata": {
    "game_id": "steam",
    "policy_id": "steam-web-v1",
    "rule_id": "steam-store-tcp-443",
    "network": "tcp",
    "process_name": "steamwebhelper.exe",
    "process_path_hash": "optional-sha256",
    "client_config_revision": "20260622.1",
    "capture_mode": "process",
    "trace_id": "optional-client-generated-id"
  }
}
```

UDP 示例：

```json
{
  "type": "OPEN_UDP",
  "target_host": "203.0.113.10",
  "target_port": 27015,
  "metadata": {
    "game_id": "example_game",
    "policy_id": "example-game-realtime-v1",
    "rule_id": "example-game-udp-27000-27999",
    "network": "udp",
    "process_name": "game.exe",
    "client_config_revision": "20260622.1",
    "capture_mode": "process"
  }
}
```

节点会校验：

```text
token 是否有效
token 是否允许 metadata.game_id
token 是否允许 metadata.policy_id
metadata.rule_id 是否属于该 policy_id
metadata.network 是否和 OPEN 类型一致
metadata.client_config_revision 是否和 token.config_revision 一致
target_host + target_port 是否命中节点本地 route_policies
```

## 七、当前不应下发给节点的字段

以下字段可以保留在业务后台或客户端配置里，但节点当前不直接消费：

| 字段 | 处理建议 |
| --- | --- |
| `GM_Game.GameName` / 图片 / 封面 / 背景图 | 客户端 UI 展示 |
| `GM_Game.StartUrl` | 客户端启动游戏 |
| `GM_GameTypeRel` / `GM_GamePlatformRel` | 客户端分类、平台筛选 |
| `ZoneName` / `Sort` / `Remark` | 客户端展示或后台管理 |
| `Hosts` / `ClearHosts` | 客户端本机 hosts 操作 |
| `SmartDns` | 客户端本机 DNS 处理 |
| `CmdExec` | 客户端本机命令 |
| `FakePingValue` | 客户端本机展示或 hook |
| `SkipMainlandIP` | 客户端本机规则判断或后台预处理 |
| `BandwidthQuality` | 后台或客户端选择节点 |
| `TelecomIP` / `MobileIP` / `UnicomIP` | 后台或客户端先选最终入口 |
| `RelayServerIP` / `RelayServerPort` | 当前节点协议还没有 relay 字段 |

## 八、业务后台需要确认的问题

1. `GM_GameID` / `GM_GameZoneID` 是否直接作为 `game_id` / `policy_id` 使用。如果使用雪花 ID，必须全程字符串。
2. `Version` 是每个区服一个版本，还是后台统一生成全局 `revision`。推荐同一批客户端配置、token、节点策略使用同一个 `revision`。
3. `Port` 里的 `0-65535` 是否能在保存或生成策略时自动改成 `1-65535`。
4. 域名规则是否存在 `api*.example.com` 这种中间通配。如果存在，当前节点无法直接表达。
5. `DisableQuic=true` 的节点是否会被后台或客户端过滤。
6. 三网 IP 是业务后台选择最终入口，还是客户端按本机运营商选择。无论哪边选，最终只把一个 `endpoint_host` 给 QUIC 客户端。
7. 中继字段是否近期要接入。如果要接入，需要单独设计中继协议，不能复用当前节点直连字段。
8. `IsAcc=false` 是否会被客户端当成排除/direct/block，而不是命中后继续发起节点 `OPEN`。
9. token 的有效期、最大连接数、限速和套餐映射是否由业务后台统一控制。
10. 节点 `hmac_secret` 是否已经能在业务后台生成、保存、轮换，并同步给控制面板。

## 九、建议接入顺序

1. 业务后台先补节点 `hmac_secret` 生成和保存。
2. 调用控制面板 `/api/backend/nodes` 同步节点，确认 `hmac_secret_configured=true`。
3. 调用 `/api/backend/token-defaults` 读取默认套餐档位。
4. 业务后台实现客户端 token 签发接口。
5. 把 `GM_GameZone` 的 `DomainRules` / `IpRules` 转换成 `route_policies_yaml`。
6. 调用 `/api/backend/policy-revisions/validate` 校验策略。
7. 校验通过后保存策略，并设置节点 `desired_policy_revision`。
8. 等待 `/api/backend/nodes/{node_id}/sync-status` 返回 `policy_state=synced`。
9. 客户端按配置命中规则，带 metadata 发起 `OPEN_TCP` / `OPEN_UDP`。

## 十、给业务后台的简短结论

现有数据库字段可以保留，但需要补一层转换：

```text
GM_Game / GM_GameZone / GM_AccNode
  -> 客户端配置
  -> 客户端短期 JWT token
  -> 节点 route_policies
  -> 控制面板节点同步数据
```

不要把数据库字段原样下发给节点。节点只认 token、OPEN 请求 metadata、target_host、target_port 和本地 `route_policies`。
